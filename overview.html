Introduction

Influish Backend is the central orchestration layer connecting all interfaces in the Influish product pipeline. It manages APIs consumed by the cross-platform Flutter application and the internal React operations panel, and supports WebSocket protocols to enable instantaneous, persistent messaging features.

By isolating business logic into highly modular services, our backend enables rapid iteration on campaign tools, KYC flows, and payment triggers, without compromising scalability or runtime speed.

Mobile App Integration

Supports JSON/REST endpoints consumed by the Flutter application to stream campaign listings, withdraw payouts, and verify social profiles.

Admin Panel Integration

Direct database gateways feeding statistical aggregations, dispute details, KYC file review boards, and transaction audit trails to the React operations dashboard.

Tech Stack

We leverage a modern, lightweight, highly active Node.js and TypeScript environment to ensure complete type safety from controller request interfaces down to database schema representations.

Core Engine

  • NestJS & Express: The core framework providing structural boundaries, automatic module loading, dependency injection, and HTTP route binding.
  • TypeScript: Strongly-typed compiler variables representing models, preventing runtime casting mismatches.

Data & Memory

  • PostgreSQL: Highly reliable, relational ACID database handling transaction registers and user balances.
  • Prisma ORM: Next-generation database mapping toolkit, generating exact TypeScript interfaces directly from sql migration schemas.
  • Redis & BullMQ: High-performance in-memory cache and non-blocking job coordinator representing robust asynchronous background pipelines.

Integration Channels

  • AWS S3 & Firebase: Decentralized cloud file storage buckets and OAuth verification systems.
  • OpenAI: LLM integrations optimizing text content extraction and brief drafts.
  • Razorpay & Cashfree: Bank clearing gateways delivering real-time payout capabilities.

Design Patterns

To maintain strict structural modularity and prevent complex domain coupling as the platform grows, we mandate specific object-oriented design standards across all modules:

  • Domain-Driven Design (DDD): Code is segregated into independent folder boundaries mapping to business domains (e.g. /auth, /campaign, /wallet). Database entities and logical controllers do not cross domain files directly.
  • Clean Architecture: Separation of core layers ensures database structures (Prisma clients) can be modified without altering primary controller validations or route definitions.
  • CQRS (Command Query Responsibility Segregation): Separation of write-side transaction inputs (Commands) from read-side dashboards queries (Queries) to maximize system scalability under heavy query traffic.
  • Event-Driven Pipelines: State transitions (e.g., campaign creation or payment success) publish event hooks processed asynchronously by Redis-backed workers, preventing system request stalling.

Code Quality & CI/CD

A robust suite of Git checking tools guarantees that no unformatted, non-compiling, or untested source files are ever integrated into the core branches.

Husky Git Hooks

Runs pre-commit checking rules blocking files if basic ESLint checks, Prettier style guide definitions, or TSC compilations encounter problems.

Conventional Commit Validation

Mandates standardized, readable descriptions (e.g. feat(wallet): trigger payout webhook or fix(chat): handle socket disconnect) to auto-generate release changes logs.
