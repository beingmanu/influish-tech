Overview & Features

The Influish Mobile App is a high-performance cross-platform Flutter application powering brand and influencer connections. By adopting a unified client repository compiled for both iOS and Android, our code guarantees synchronized features and pixel-perfect design alignment.

The application runs two distinct user workflows inside a single build package:

For Influencers

  • Campaign Discovery: Filter niche categories and apply directly to matching campaign boards.
  • Metrics Tracker: View views, clicks, engagement metrics, and payment summaries.
  • Wallet Ledger: Initiate withdrawals, audit payouts, and track bank reconciliations.
  • Identity Upload: Complete secure profile verification with image-croppers.

For Brands

  • Campaign Builder: Fast form templates with inline AI assistance mapping briefs.
  • Creator Review Board: Approve applied content uploads or negotiate revisions.
  • Instant Transactions: Process campaigns locks via Razorpay/Cashfree gates.
  • Aggregated Analytics: Track click conversions and ROI charts.
Common Utilities
Both systems share real-time Socket.IO chat windows, background notifications via FCM, offline caches (Floor SQLite), and multi-language routing definitions (GoRouter).

Installation & Setup

New developers must follow these consecutive terminal commands to configure and compile the mobile client:

1. Dependencies Installation

Navigate into the project directory and retrieve package listings:

Terminal (Bash) 
cd influish/frontend_app
flutter pub get

2. Run Code Generation

Trigger build_runner to parse abstract Retrofit endpoints and Floor databases:

Terminal (Bash) 
flutter packages pub run build_runner build --delete-conflicting-outputs
Code Generation Requirement
Any structural model updates or API endpoint modifications mandate running the generator. Keep `watch` active in a background terminal during active code changes:
flutter packages pub run build_runner watch

3. Compile & Run

Launch development compilation targeting local devices:

Terminal (Bash) 
# Android Emulator or Physical Device
flutter run --flavor dev -t lib/main.dart

# iOS Simulator or Physical Device  
flutter run --flavor dev -t lib/main.dart -d ios

Folder Architecture

The project code structure segregates layers into features and domains, keeping layout implementations fully separate from data operations. Here is a high-level visual map:

lib/
├── main.dart // App launch entry point
├── app_root.dart // App shell, routers and themes load
└── src/
├── config/ // Navigation structures and CSS themes
├── utils/ // Share color schemes, local data keys and extensions
├── common/ // Shared UI cards, common loaders, and generic network models
│ ├── blocs/ // Global session blocs (e.g. auth status, balances)
│ └── widgets/ // Global reusable button components
├── presentation/ // Graphical UI screens grouped by roles
│ ├── influencer/ // Influencer explore tab, details and wallet views
│ └── brand/ // Brand campaigns setup panel
├── domain/ // Pure interfaces, domain models and entities
│ ├── influencer/ // Core data models for influencer flows
│ └── brand/ // Campaign structure models
└── data/ // Repository implementations and SQLite local data controllers
├── influencer/ // Remote network calls for influencers
└── brand/ // Brand remote campaigns interfaces
  • Domain Layer: Holds raw model representations (no network or json serialization annotations). This represents pure business guidelines.
  • Data Layer: Houses network files, Retrofit endpoints, JSON deserialization logic, and database caches.
  • Presentation Layer: Houses views, responsive widgets, and logic blocks mapping local states (e.g. StateNotifier or BLoC).

Coding Guidelines

To maintain high code clarity and minimize merge disputes, all developers must adhere to these coding paradigms:

  • Strong Type Checking: Do not use dynamic typings. Declare type outputs for all API payloads, routes parameters, and state notifier actions.
  • Formatting: Run `dart format .` before staging changes. Unformatted layout code will be rejected during automatic pre-commit runs.
  • Resource Constraints: Wrap responsive cards inside ScreenUtil definitions (e.g. .w, .h, .sp) to prevent pixel clippings on compact displays.
  • Optimizations: Mark static UI blocks with const constructors. Use image caches for external network avatars to prevent repetitive resource fetches.

Contributing Guide

All developers push custom code branches under a structured, peer-reviewed Git setup:

1. Feature Branches

Create small features branches with descriptive handles:
git checkout -b feature/campaign-ai-generator

2. Local Linting Runs

Review lint warnings locally before staging:
flutter analyze && flutter test

3. Peer Review Integration

Open a Pull Request with a clear changelog description. Target branches require a minimum of two peer reviews before master branch integration.
